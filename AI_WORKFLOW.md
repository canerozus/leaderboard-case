# AI_WORKFLOW.md


## Tools

- **Claude Code** (Opus 4.7, 1M context window) — primary AI surface. CLI in terminal, persistent memory across chat sessions.
- **`superpowers` skill set** for Claude Code — process skills used at each stage: `brainstorming`, `writing-plans`, `subagent-driven-development`, `requesting-code-review`, `receiving-code-review`, `verification-before-completion`.

## Flow used for this case

```
brainstorming → DESIGN.md → writing-plans (×3) → subagent-driven execution (per plan)
                                                  → requesting-code-review
                                                  → fix with receiving-code-review
                                                  → verification-before-completion
                                                  → merge
```

1. **Brainstorm** (`superpowers:brainstorming`). One chat to evaluate the case requirements: scale (10M registered / 2M DAU), the "instant rank" UX, the "see your own rank with 3 above + 2 below" requirement, weekly reset + 2% prize pool semantics, what's in scope vs not (no anti-cheat beyond rate limit, no friends/social, no multi-region). Output: a set of explicit trade-off questions — Mongo-or-Redis-as-primary, polling-or-SSE, monorepo-or-separate-projects, fail-open semantics.
2. **Design.** Wrote `DESIGN.md` — architecture, data model, fail-open contract, project structure, deploy topology, scale-out story.

3. **Plan** (`superpowers:writing-plans`). Three implementation plans, each generated in one pass:
   - **Plan 1 — Backend** (44 tasks, 10 phases) → branch `feat/backend-api`
   - **Plan 2 — Frontend** (31 tasks, 8 phases) → branch `feat/frontend-spa`
   - **Plan 3 — Deploy + Docs** (22 tasks, 2 slices) → branches `docs/case-deliverables` and `infra/prod-deploy`

   Each task is single-PR-sized; TDD prescribed for risk areas (CacheService fail-open contract, prize-distribution math, weekId boundary, JWT). Each plan carries a shared **Execution coordination** block at the top: stage-based timeline, branch ↔ plan mapping, cross-plan unlock gates marked inline at each phase boundary, plan-status table that fresh chats know to update.
4. **Execute** (`superpowers:subagent-driven-development`). One chat per plan, one subagent per task (~97 dispatches total). Each subagent receives only its task's section as prompt; the orchestrator chat reviews each diff and ticks `- [x]` before the next dispatch. Keeps the orchestrator's context clean.
5. **Review and verify before merge.** Same invariant after every plan, regardless of which planning flow was used:
   - `superpowers:requesting-code-review` — second-opinion sweep on the branch before opening a PR.
   - `superpowers:receiving-code-review` — adopt or push back with technical reasoning; no performative agreement.
   - `superpowers:verification-before-completion` — actually run the tests, hit the endpoints, check the deployed URL before claiming "done."
6. **Merge** in stage order: `feat/backend-api` → `docs/case-deliverables` → `feat/frontend-spa` → `infra/prod-deploy` → tag `v1.0` → email HR.

## What I deliberately skipped (and why)

**The full BMAD architecture-then-epics-then-stories flow.** My normal pipeline is:

```
brainstorming → bmad-create-architecture → bmad-create-epics → bmad-create-stories
              → (per story: brainstorming → writing-plan → executing-plan)
              → review → verification
```

For this case I cut the middle ceremony:

- `DESIGN.md` already does what `bmad-create-architecture` would produce (data model, topology, role split, fail-open contract, deploy story). Adding a separate ARCHITECTURE.md generated by the BMAD step would have been near-duplicate content.
- The `writing-plans` output was already at story granularity (single-PR-sized tasks with verification per task). Re-cutting it through `bmad-create-epics` + `bmad-create-stories` would have re-emitted the same tasks under a different folder shape at extra orchestration cost.
- Per-story brainstorming passes were unnecessary when the DESIGN.md spec already carries the design decisions; the plans are mechanical translations of that spec.

**Trade-off I'm aware of:** without per-story brainstorm I lose the "did I think hard enough about THIS step" gate. Mitigated by (a) the level of detail in DESIGN.md, (b) per-task code review on the merge path, (c) the fact that `writing-plans` itself prescribes TDD on the highest-risk tasks.

## What I'd do at larger scope

The fuller BMAD flow above. The benefit isn't better code at this scope — it's better-checked decisions per increment, which matters more on long-running products than on a single deliverable. The post-execution invariant (`requesting-code-review` → fix → `verification-before-completion`) is the same either way.

## Where I made the call

### Architecture (DESIGN.md is the canonical record)

- **DB role split:** MongoDB as the **primary score store** (Bucket Pattern, daily partition), Postgres for identity + weekly_history + payouts, Redis as a **fail-open cache only**. Mongo is source of truth on any disagreement.
- **Fail-open contract:** every Redis call lives in a single `CacheService`, wrapped in `try/catch` → log + return `null`; never throws. Feature services treat `null` as "cache unavailable, do the slow thing."
- **Lazy single-flight rehydration** on read miss instead of a recovery daemon.
- **API split:** `/leaderboard/top` vs `/leaderboard/me` instead of one polymorphic endpoint — each call is the minimum payload for one screen state.
- **Single Docker image, two entrypoints** (`index.api.ts`, `index.worker.ts`) — same codebase, two `command:` overrides in compose.
- **1-second TTL on top-100 read cache** instead of invalidate-on-write — bounds the worst-case staleness without an explicit invalidation path on every write.
- **No idempotency on `/score/submit`** (case-scope decision; documented).
- **Top-1000-per-week history cap** in Postgres weekly_history.
- **Moved demo-traffic generator** out of the worker into a CLI seed script — keeps the production worker single-purpose (cron only).
- **Polling (5–7s)** instead of SSE — simpler, fits the case scope, fewer moving parts at the edge.

### Process

- **Skipping BMAD ceremony** for this scope (above).
- **Single root `.env`** consumed by backend, Vite (`envDir: '..'`), and docker-compose (`env_file: ../.env`) — rejected per-package envs after starting one and consolidating.
- **Plan-per-branch with Conventional Commits naming** (`feat/backend-api`, `feat/frontend-spa`, `docs/case-deliverables`, `infra/prod-deploy`). One chat = one branch = one PR. Per-task or per-phase branching would have produced ~97 PRs — drowning.
- **Stage-based (sequence-based, not calendar-based) cross-plan coordination markers** in plan files — gates trigger on phase completion, not on wall clock.
- **`docs/` is gitignored** — the implementation plans (`docs/superpowers/plans/*.md`) are local-only artifacts for the executing agents; the public repo carries only DESIGN.md, AI_WORKFLOW.md, PRD.md, README.md, and code.

## Honest notes

- Claude is fast at scaffolding and docs; it is **not** load-bearing for design judgment. Every architectural call above was made by me, with AI as a sounding board.
- The plans were generated in one `writing-plans` pass and then **audited structurally** before commitment — three Explore subagents in parallel reported back on phase structure, task granularity, dependencies, and cross-plan touchpoints. The audit's verdict ("execute as-is") is what made me skip the BMAD re-cut.
- Parallelism in execution: `docs/case-deliverables` and `feat/backend-api` ran on overlapping branches; FE waited for BE Phase 7 (the only hard cross-branch gate); deploy was the linear final stage.
- The `.env` story illustrates the workflow well — I started with per-package envs, pushed back to consolidate to one root env, **and saved the preference as a memory** so a future session in this project starts from the corrected default.
